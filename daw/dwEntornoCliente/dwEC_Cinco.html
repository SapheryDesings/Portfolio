Desarrollo web en Entorno de Cliente Principal

Tema 5. - Integración al código de la gestión de eventos y formularios

La interacción entre usuario y aplicación web consiste en que el usuario utiliza mecanismos de entrada, y la aplicación muestra la información que debe ser tratada.

Existen eventos que materializan los cambios que realiza el usuario sobre un determinado objeto. A estos cambios se los conoce como eventos.

5.1.- Modelo de gestión de eventos

Cuando un usuario realiza un cambio sobre una página web, se accionan los eventos.

El DOM es el encargado de crear la jerarquía entre estos objetos, es el encargado de gestionar los eventos.

Los eventos se encuentran situados en componentes de la página HTML, es posible desencadenar una acción interactuando con un botón de la página. Para controlar un determinado evento, necesitamos un manejador, podría ser la palabra onClick. Desencadenará la función que le asociemos en cuanto hagamos clic en el elemento que lo contenga.

Una vez asociado el manejador al elemento, necesitamos desencadenar alguna acción. Con el cual podemos referirnos a los objetos del DOM sin necesidad de recargar la página.

                
                    < img src=”Ilerna.jpeg” onclick=”alert(‘Click en la
                    imagen’);”>

5.1.1. Eventos de ratón

Estos eventos se activan en el momento en que el usuario hace uso del ratón para realizar una determinada acción. Cualquier movimiento de la flecha del ratón o acción sobre algunos de sus botones puede desencadenar un evento.

Evento Manejador Descripción
Click onclick Este evento se produce cuando pulsamos sobre el botón izquierdo del ratón
Dblclick ondblclick Este evento se produce cuando hacemos doble clic sobre el botón iquierdo del ratón
Mousedown onmousedown Este evento se produce cuando pulsamos un botón del ratón
Mouseout onmouseout Este evento se produce cuando el puntero del ratón está enciam de un elemento y es desplazado fuera de este
Mouseover onmouseover Este evento se produce cuando el puntero está fuera de un elemento y es desplazado dentro de este
Mouseup onmouseup Este evento se produce cuando soltamos el botón del ratón que previamente habíamos pulsado
Mousemove onmousemove Este evento se produce cuando el puntero del ratón está dentro de un elemento

5.1.2. Eventos de teclado

Estos eventos se originan cuando el usuario pulsa alguna tecla del teclado

Evento Manejador Descripción
Keydown onkeydown Este evento se produce cuando pulsamos alguna tecla del teclado
Keypress onkeypress Este evento se produce si pulsamos una tecla de un carácter alfanumérico
Keyup onkeyup Este evento se produce cuando soltamos una tecla

5.1.3. Eventos HTML

Estos eventos se producen cuando se origina un cambio en la página del navegador. También pueden ocurrir cuando existe alguna interacción entre el cliente y el servidor.

Evento Manejador Descripción
Load onload El evento Load hace referencia a la carga de diferentes partes de la página. Este actúa sobre el objeto Window cuando la página se ha cargado completamente. En el objeto < img> se produce cuando se ha cargado la imagen, y en el elemento < object> se acciona al cargar el objeto por completo
Unload onunload El evento Unload actúa sobre el objeto Window cuando la página ha desaparecido por completo. También se acciona en el elemento < object> cuando desaparece el objeto
Abort onabort Este evento se produce en el momento en que el usuario detiene la descarga de un elemento antes de que haya finalizado, actúa sobre un elemento < object>
Error onerror El evento Error actúa sobre un objeto Window cuando se produce un error JavaScript, en el elemento < img> en el caso de que la imagen no se cargue y en el elemento < object> en el caso de que no se cargue correctamente.
Select onselect Se acciona cuando seleccionamos textos de las cajas de texto (< input> o < textarea>).
Change onchange El evento se produce cuando los cuadros de texto < input> y < textarea> pierden el foco y el contenido que tenían ha variado. También se produce cuando el elemento < select> cambia de valor.
Submit onsubmit Este evento se produce cuando pulsamos sobre un botón tipo submit
Reset onreset Este evento se produce cuando pulsamos sobre un botón tipo reset
Resize onresize Este evento se produce cuando redimensionamos el navegador, actúa sobre el objeto Window
Scroll onscroll Se produce cuando varía la posición de la barra de desplazamiento o scroll en cualquier elemento que la tenga
Focus onfocus Este evento se produce cuando un elemento tiene el foco
Blur onblur Este evento se produce cuando un elemento pierde el foco

5.1.4. Eventos DOM

Los eventos DOM, o eventos de mutación, son los que se originan cuando existe algún cambio en la estructura DOM de la página.

Evento Descripción
DOMSubtreeModified Este evento se produce cuando se añaden o se eliminan nodos en el subárbol de un elemento o documento
DOMNodeInserted Este evento se produce cuando añadimos un nodo hijo a un nodo padre
DOMNodeRemoved Este evento se produce cuando eliminamos un nodo que tiene nodo padre
DOMNodeRemovedFronDocument Este evento se produce cuando eliminamos un nodo del documento
DOMNodeInsertedIntoDocument Este evento se produce cuando añadimos un nodo al documento

5.2.- Captura y uso de eventos

Llegado el momento de la integración de códigos, es muy importante tener en cuenta que, si no existen eventos, no hay scripts.

En la mayoría de las páginas existen una serie de eventos ya programados que van a disparar la ejecución de estos scripts. Esto sucede por que fue diseñado para incluir interactividad en las distintas páginas cuando el usuario realiza una acción determinada en la página y esta reacciona.

En el momento en que el usuario realiza alguna acción, se produce un determinado evento. Existe una serie de eventos que no están relacionados de forma directa con las distintas acciones del usuario.

La técnica de gestión de eventos pueda variar con tal de hacer un código más independiente de la estructura HTML.

En versiones anteriores, los eventos (interacciones entre usuario y sistema) eran capturados en casi todos los casos por gestores de eventos definidos como atributos en etiquetas HTML.

Actualmente, este modelo sigue en activo, aunque los navegadores incorporan un modelo propio, que permite que se puedan programar respuestas más inteligentes a las distintas interacciones que puede tener el usuario con los objetos que formen parte del documento.

5.2.1. Modelo de registros de eventos en línea

En el modelo de registros de eventos en línea, se puede añadir un determinado evento como si fuera un atributo más a la etiqueta HTML correspondiente.

                
                    function alertar(){
                        alert(“Ha pulsado el enlace”);
                    }

En el lenguaje HTML es posible utilizar caracteres en mayúsculas y minúsculas en los nombre de los eventos onClick y onMouseOver. Sin embargo, en XHTML los atributos deben ir obligatoriamente siempre en minúsculas: debemos escribir onclick y onmouseover.

5.2.2. Modelo de registro de eventos tradicional

Los navegadores antiguos utilizaban el modelo en línea, en el modelo de registro tradicional, el evento pasa a convertirse en una propiedad del elemento.

Sigue siendo válida hoy en día, su principal ventaja es que permite asignar un determinado evento a un objeto, separa el código de la estructura. Los nombres de los eventos se escriben siempre en minúsculas.

Si deseamos eliminar un gestor de eventos de un determinado elemento, asignarle el valor de null.

Otra de sus ventajas es que el gestor de eventos es una función, es posible realizar una llamada directa a ese gestor, de modo que podemos disparar el evento de manera manual.

Sin paréntesis

En el registro de eventos no utilizamos paréntesis(). Cabe destacar que este no es el procedimiento ideal, porque queremos que se pueda ejecutar una función al disparar un determinado evento.

Uso de la palabra reservada This

El uso de esta palabra reservada en el modelo de línea es bastante frecuente cuando deseamos programar un gestor de eventos.

5.2.3. Modelo de registro avanzado de eventos W3C

En la especificación del DOM de nivel 2, cuenta con una forma sencilla de poder dar registro a los distintos eventos que deseemos sobre un determinado evento.

Es fundamental utilizar el método addEventListener().

Este método cuenta con tres argumentos diferentes: el tipo de evento, la función que se desea ejecutar y un valor booleano que puede ser true o false y que va a servir para indicar si podemos capturar o no un evento.

Entre las ventajas de este método, destaca que permite añadir los eventos que deseemos.

                
                    document.getElementById(“mienlaceuno”).addEvent-
                    Listener(‘click’, alertar,false);
                    document.getElementById(“mienlaceuno”).addEvent-
                    Listener(‘click’, avisar,false);
                    document.getElementById(“mienlaceuno”).addEvent-
                    Listener(‘click’, chequear,false);

Palabra reservada This

La palabra reservada This se comporta igual que en el modelo tradicional.

Entre los inconvenientes, debemos señalar que no es posible saber con antelación cuáles son los eventos que han sido registrados en un elemento.

En el modelo tradicional, podemos hacer lo siguiente.

                
                    alert(elemento.onclick)

Este método va a devolver el valor undefined siempre que no existan funciones registradas en ese determinado evento, o bien el nombre correspondiente a la función registrada para ese evento, aunque en este modelo no tenemos esa opción.

Nivel 3 del DOM ya introdujo un nuevo método denominado enetListenerList que permite almacenar una lista de las distintas funciones que se han asignado a un determinado elemento. No lo soportan todos los navegadores.

Una vez eliminado un determinado evento, podemos utilizar el método:

                
                    removeEventListener(): elemento.removeEventListener(‘
                    evento’, función, false|true);

Si deseamos cancelar un evento, usaremos el método preventDefault().

5.2.4. Modelo de registro de eventos según Microsoft

Microsoft también ha desarrollado un modelo de registro de eventos.

  • Cuando deseemos registrar un evento, es necesario que lo enlacemos
                
                    attachEvent(): elemento.attachEvent(‘onclick’, hacerAlgo);
  • Si, se necesitan dos gestores distintos de un mismo evento
                
                    elemento.attachEvent(‘onclick’,hacerUnaCosa);
                    elemento.attachEvent(‘onclick’, hacerOtraCosa);
  • En el caso de que deseemos eliminar un determinado evento
                
                    detachEvent(): elemento.detachEvent(‘onclick’, hacerAlgo);

Si comparamos este método

  • No es posible capturar los eventos porque siempre van a burbujear
  • Al estar referenciada en vez de copiada la función que gestiona el evento, la palabra reservada This siempre va a hacer referencia a Window, por lo que va a resultar completamente inútil

Según estas dos debilidades, cuando un evento esté burbujeando hacia arriba, no podremos conocer qué elemento HTML ha gestionado ese evento.

5.2.5. Orden de disparo de los eventos

En el supuesto de que tengamos un elemento que se encuentre contenido dentro de otro elemento, deberemos programar el mismo tipo de evento para ambos.

Tenemos dos modelos diferentes

  • Netscape afirmó que el evento en el elemento 1 va a suceder primero, denominándolo captura de eventos.
  • Sin embargo, Microsoft mantenía que el evento en el elemento 2 tiene precedencia ante el otro. A este hecho se lo denomina burbujeo de eventos.

Los dos modelos son totalmente opuestos.

5.2.6. Modelo W3C

Este modelo se decanta por una posición intermedia en la que se supone que, al producirse un evento en su modelo de eventos, primero se producirá una fase de captura hasta que consiga llegar al elemento de destino.

Ya se puede producir la fase de burbujeo hacia arriba. Este modelo está definido como estándar, por lo que todos los navegadores lo deberían seguir para ser compatibles entre ellos.

5.3.- Utilización de formularios desde código

Un formulario web es una herramienta que se utiliza para enviar, recibir o gestionar datos que se transmiten entre el cliente y un servidor web. Cada uno de los elementos se utiliza para almacenar un tipo de dato o para activar funcionalidades del formulario.

Los formularios están enmarcados en HTML y su función es dar la posibilidad a los usuarios para que interactúen con los datos, convirtiendo una simple página web en una aplicación web.

Estructura de un formulario

Los formularios se definen con etiquetas. Dentro de cada una de las etiquetas, es posible acceder a su atributo.

La etiqueta principal del formulario es form, la cual indica el comienzo y el final del formulario.

Para hacer que el formulario sea funcional, necesita inicializar dos atributos:

  • action: donde se indicará la URL para el envío de los datos
  • method: indica el método de envío de los datos. Existen dos opciones, POST o GET. En el caso de POST, los datos se enviarán ocultos, mientras que GET hace que los datos viajen a través de la URL.

Nos podemos encontrar son enctype, que define el tipo de codificación que utilizaremos a la hora de enviar el formulario, y el atributo accept, que indica el tipo de archivos que acepta el servidor.

Dentro de las etiquetas de body se han introducido las etiquetas form. En el atributo action, le indicamos que los datos del formulario se enviarán al archivo form.php y su método de envío es POST.

Elementos de un formulario

Para indicar este formulario qué datos queremos enviar y cuándo los enviaremos, utilizaremos los elementos de formulario. El primer elemento que nos encontramos es el input. A estos inputs se los conoce como controles de formulario y campos de formulario. Son los encargados de almacenar los datos que vamos a enviar en el formulario.

Atributos de los que dispone input.

Estructura de la etiqueta input

Al tratarse de una etiqueta unaria, se utiliza la barra / para finalizarla.

  • Type: nos indica el tipo de elemento que se va a definir. Los valores que acepta este atributo son: text (cuadro de texto), password (cuadro de contraseñas, ocultando los valores introducidos), checkbox (casillas de verificación), radio (opciones entre valores), submit (botón de envío de formulario), reset (botón que vacía los campos), file (botón de imagen de formulario), button (botón del formulario)
  • Name: asigna un nombre al elemento. Si no se le asigna un nombre al elemento, el servidor no lo podrá identificar y no podrá acceder a él
  • Value: inicializa el valor del elemento
  • Size: tamaño inicial del elemento. En los campos de text y password, hace referencia al número de caracteres
  • Maxlength: indica el número máximo de caracteres. Se utiliza para los campos de text y password
  • Checked: elemento exclusivo para los elementos checbox y radio. Se indica que para definir qué elemento queremos seleccionar
  • Disable: deshabilita el elemento. Sus datos no serán enviados al servidor
  • Readonly: evita la modificación del valor del elemento
  • Src: permite asignar una URL donde se encontrará la imagen del botón del formulario
  • Alt: descripción del elemento. Al posicionar el ratón encima del elemento, nos mostrará un texto alternativo

Tipos de input

El objetivo de los tipos de input es facilitar la gestión de los datos (envío, recepción y manejo) a través del formulario. Tipos de input:

  • Cuadro de texto: se muestra un cuadro de texto en el que el usuario podrá introducir los datos en formato texto. Se tiene que indicar que es un campo de texto: type="text"
  • Cuadro de contraseña: es similar al cuadro de texto, con la diferencia de que los caracteres introducidos por parte del usuario no se muestran por pantalla. Se utiliza especialmente para introducir contraseñas. Es necesario indicarlo de la siguiente forma: type="password"
  • Casillas de verificación: permiten al usuario seleccionar diferentes opciones. Se indicarán de la siguiente forma: type="checkbox"
  • Opciones de radio: este tipo de elemento permite seleccionar una serie de opciones excluyentes entre sí, es decir, solamente permite seleccionar una opción entre todas las opciones disponibles. Se indica de la siguiente forma: type="radio"
  • Botón de envío: este elemento se encarga de enviar los datos del formulario al servidor. Se indicará de la siguiente forma: type="submit"
  • Botón de reset: sirve para vaciar los campos del formulario. Se indica con type="reset"
  • Ficheros adjuntos: este elemento permite adjuntar archivos al formulario. Se deberá implementar un control de este elemento, ya que no se han indicado las limitaciones de los archivos de forma predeterminada. Se indica mediante type="file"

Manipulación de formularios

Podemos acceder a un formulario puesto que cuenta con una serie de características y métodos del objeto Document. Podemos acceder a través del atributo id.

                
                    //Sintaxis
                    var miFormulario = document.getElementById(‘FormularioDatos’);

El método getElementByTagName() permite buscar a través de la etiqueta del formulario.

                
                    //Sintaxis
                    var formularios = document.getElementByTagName(‘form’);

Nos permite acceder al elemento que estemos buscando, accederemos a la colección de formularios del objeto Document. Accederemos al formulario a través del índice o del nombre.

                
                    //Para acceder a través del índice:
                    var miFormulario = document.getElementById(‘formularioDatos’);
                    //Para acceder a través del nombre
                    var otroFormulario = document.forms[‘infoUsuario’];

Envío de formularios

Enviar un formulario, debemos crear una colección de valores de los diferentes elementos que componen el formulario mediante una colección clave:valor. La clave nos permitirá acceder a través del nombre a una determinada colección.

También existe la posibilidad de realizarlo a través del índice. El acceso por índice es bastante más complejo.

Si hacemos referencia a los métodos de envío, disponemos del método GET, que combina todos los pares clave:valor en el interior de la URL de la petición. Presenta un gran inconveniente: cuando se realiza algún tipo de envío, se pone en peligro la seguridad de los datos que viajan por la URL del navegador.

El método POST almacena en el interior del cuerpo la petición realizada con los pares clave-valor, de esta forma se evita poner en riesgo la seguridad que se comenta con el método de envío GET.

Desde el punto de vista del servidor, una vez que se recibe el formulario con todos los datos, se pueden consultar a través del objeto HttpRequest.

Podemos llevar a cabo el envío de formularios a través de un control HTML tipo submit, como pueden ser las etiquetas input o button, con un botón tipo imagen, o bien con un atributo submit.

Se comenzará realizado el envío a una dirección especificada en el atributo action del formulario. Si nos encontramos con algún caso en el que el formulario no especifique la URL en su atributo action, consideraremos que el envío se va a realizar a la URL que corresponda a la página actual.

Envío mediante código

Para realizar el envío mediante código, utilizaremos el método submit() correspondiente al objeto que va a representar dicho formulario.

Deberemos utilizar alguno de estos métodos para conseguir la referencia de un formulario determinado. Una vez que la tengamos, podremos utilizar el método submit del objeto formulario para que se realice el envío del formulario sin que sea necesario añadir los elementos HTML en el propio documento.

                
                    function envio{
                        document.forms[“formularioDatosEmpresa”].submit();
                    }

Validación de envío del formulario

En las aplicaciones de carga es frecuente añadir una serie de reglas de validación en los datos que se introduzcan para evitar que se produzcan peticiones erróneas y así conseguir optimizar la red.

Mediante el control del evento onsubmit podemos controlar los diferentes datos antes de que sean enviados.

Podemos definir este evento como un atributo dentro de la etiqueta form o creando un manejador a través de la instrucción addEventListener.

Los datos introducidos no sean los correctos, o si comprobamos que alguna de las reglas no se está cumpliendo, podemos cancelar el proceso de envío utilizando el método preventDefault.

Restablecer datos del formulario

Es conveniente restablecer el formulario con los datos que utilizamos inicialmente. El método reset(), o podemos introducir botones HTML tipo reset en el propio objeto formulario.

5.5.- Validación y envío

El momento de validar el formulario es un proceso que debe estar centrado en revisar un formulario para comprobar que todos los datos que se han introducido lo estén en forma correcta.

Si un formulario cuenta con un campo de tipo texto en el que debemos escribir un email, es conveniente que comprobemos que ese email está escrito correctamente antes de continuar.

Existen dos métodos:

  • Uno en el lado del servidor, mediante la utilización de scripts CGI, PHP, ASP, etcétera.
  • Otro en el lado del cliente, que utiliza generalmente JavaScript

Es bastante más seguro el método utilizado en el lado del servidor, algunas veces también resulta más complicado de programar. La validación en el lado del cliente es más sencilla y rápida de realizar, ya que no es necesario que el navegador se conecte al servidor para poder validar el formulario, el usuario puede recibir la información sobre los posibles errores encontrados.

La mejor opción es enviar los datos al servidor para que este los valide correctamente. Existen ocasiones en las que los usuarios insertan una gran cantidad de datos incorrectos y llegan a sobrecargar el servidor. Para solucionar esta sobrecarga, realizaremos en el navegador un primer filtrado de los datos del cliente una vez que ha pulsado el botón enviar. Si existe algún dato incorrecto, esta información no se enviará al servidor y mostrará un feedback al usuario indicando el campo que debe modificar.

Se realiza llamando a una función que analizará si los datos introducidos cumplen con las restricciones impuestas.

Estructura de un formulario para la validación

Para indicar al usuario que debe realizar una validación de los datos antes de enviarlos al servidor, se indicará en su misma estructura. Para ello existe el evento onsubmit, que ejecutará una acción una vez que se ha llamado al action del formulario.

Para que los datos sean validados, es necesario implementar la función validarForm().

  • Validar un campo obligatorio: hay ciertos campos que son obligatorios y es necesario comprobar que el usuario los ha rellenado antes de que los datos sean enviados al servidor. En este caso, comprobaremos si existen datos introducidos en uno de los campos de texto
                
                    < script type=”javascript”>
                        function validarForm() {
                            valor = document.getElementById(“campo”).value;
                            if (valor == null || valor.lenght == 0) {
                                alert(“El campo no puede estar vacío”);
                                return false;
                            }
                            return true;
                    < /script>
  • Validar un campo de texto numérico: hay campos en los que solamente nos interesa que inserten números, como son los teléfonos, DNI(sin letra) o el código postal
                
                    < script type=”javascript”>
                        function validarForm() {
                            valor = document.getElementById(“telefono”).value;
                            if (isNaN(valor)) {
                                alert(“El campo tiene que ser numérico”);
                                return false;
                            }
                        return true;
                    < /script>
  • Validar un checkbox: en ocasiones, nos interesa conocer si un checkbox se ha seleccionado o no, como puede ser la aceptación de las codiciones de uso o de los términos legales
                
                    < script type=”javascript”>
                        function validarForm() {
                            valor = document.getElementById(“condiciones”).value;
                            if (!valor.checked) {
                                return false;
                            }
                        return true;
                    < /script>

5.6- Expresiones regulares

Las expresiones regulares son patrones de búsqueda que se pueden utilizar para encontrar un texto que coincida con un determinado patrón.

Existen distintas situaciones en las que tendremos que buscar cadenas que deben cumplir un patrón en vez de una cadena exacta.

Es más sencillo hacerlo a través de estas expresiones regulares, aunque tienen una sintaxis un poco compleja.

Podemos gestionar estan expresiones a través del objeto RegExp

                
                    var expresion = /expresión regular/;

Esta expresión regular debe ir entre barras / y no lleva comillas. Las comillas solo se utilizarán en la expresión regular cuando se forme el patrón.

Características
Carácter Coincidencias Patrón Ejemplo de cadena
^ Al inicio de una cadena /^Esto/ Coincidencia en “Esto es…”
$ Al final de la cadena /final$/ Coincidencia en “Esto es el final”
* Coincide 0 o más veces /se*/ Que la “e” aparezca 0 o más veces: “seeee” y también “se”
¿ Coincide 0 o 1 vez /ap? Que la “p” aparezca 0 o 1 vez: “apple” y “and”
+ Coincide 1 o más veces /ap+/ Que la “p” aparezca 1 o más veces: “apple”, pero no “and”
{n} Coincide exactamente n veces /ap(2)/ Que la “p” aparezca exactamente dos veces: “apple”
{n,} Coincide n o más veces /ap(2,)/ Que la “p” aparezca 2 o más veces: “apple” y “appple”, pero no en “apabullante”
{n,m} Coincide al menos n, y máximo m veces /ap{2,4}/ Que la “p” aparezca al menos 2 veces y como máximo 4 veces

El objeto RegExp

El objeto RegExp lo podemos utilizar tanto como un objeto como una expresión literal. Es posible utilizarlo creando un constructor:

                
                    var expresionregular = new RegExp(“Este es el Texto
                    Expresión Regular”);

Esta expresión RegExp literal la podemos compilar una vez que se ejecute el determinado script. Es recomendable que utilicemos el literal siempre que sepamos que la función no se va a modificar, ya que siempre es mucho más eficiente una versión compilada.

En aquellos casos en que sepamos a ciencia cierta que la expresión regular va a sufrir alguna modificación, utilizaremos el objeto.
